研究背景与问题

研究背景

在大数据时代，传统关系型数据库难以满足海量数据的存储需求，因此现代数据密集型应用多采用了键值存储系统。而主流的键值存储系统大多采用了写优化的 LSM-tree 架构。例如，Google 的 BigTable 和 LevelDB，Facebook 的 RocksDB，以及 Apache Cassandra 等。

一个典型的 LSM-tree 键值存储系统由两个组件组成：一个在内存中包含一个 MemTable 和一个 Immutable Memtable，用于作为缓存快速接受用户写入的数据；另一个在磁盘中，由多个 SSTable 文件组成，并将这些 SSTable 文件逻辑划分到多个 level 中（默认情况下每个 level 的容量是上一个 level 容量的 10 倍），其中 SSTable 是 Sorted String Table 的意思，SSTable 内的键值对是有序的，除了 level-0 之外，其他 level 中的 SSTable 之间也是有序的，也就是 SSTable 之间没有重叠的键值范围。

执行写操作时，键值对首先写到 MemTable 中，当 MemTable 写满后转为 ImmuTable MemTable。然后 ImmuTable MemTable 通过一个后台的 Minor Compaction 操作被直接顺序写到 level-0 中的一个新的 SSTable 文件中。当高层level的 SSTable 数据量达到容量限制后，后台线程会执行一个 Major Compaction 操作，该操作会对高层和低一层的 SSTable 文件进行合并，主要操作包括：读取数据；合并排序；生成新的 SSTable 文件写回低层。

问题与挑战

问题：作为写优化的数据结构，LSM-tree 在磁盘中采用了分层的方式来组织数据，这同时也带来了严重的读放大问题。
读放大是在一次读操作中用户实际读取的数据量和请求的数据量的比例。在分层的基础上，如果用户没有在内存的 MemTable 中读到请求的数据，则需要依次对每层的 SSTable 进行检查，直到获得目标数据或所有层都被检查过。除了 level-0 之外，键值对在每个 level 都是有序排列的，所以在 level-0 中需要检查多个 SSTable，而在其他 level 中只需要检查一个 SSTable。此外，在检查每个 SSTable 时还需要读取多个数据块，这样就带来了非常严重的读放大问题。

现状：为了缓解读放大问题，LSM-tree 键值存储系统多使用 Bloom Filter（BF）来加速读操作。BF 可以用于检索一个集合中是否包含某个元素。为每个 SSTable 构建一个 BF ，将 SSTable 中的键值对映射到 BF 后，通过检查 BF 便可以快速判断 SSTable 中是否包含目标 KV 对。然而在将 BF 集成到 LSM-tree 的过程中也存在一些问题：

  • BF 是保存在 SSTable 内的，为了减少读磁盘的开销，通常会将大量的 BF 缓存在内存中，这会带来很高的内存空间开销；
  • BF 不支持删除操作和动态调整容量，而每次的 Compaction 操作都会合并旧的 SSTable 文件生成新的 SSTable 文件，这导致 Compaction 后需要重新构建大量BF；
  • 此外，BF 的重建会导致已缓存的 BF 失效；

挑战：如何以更低的开销来有效缓解读放大问题，且不影响写操作的性能

研究途径与方法

新型检索结构

在过去几年中出现了一系列新的数据结构作为布隆过滤器（BF）的替代品。其中 Dynamic Cuckoo Filter 由多个同构的 Cuckoo Filter 以链表的形式链接而成，具有以下优点：

  • 支持可靠的元素删除操作以及灵活的结构扩展；
  • 与其他支持删除操作的同类数据结构相比具有更优的空间效率和更快的查询速度；

CF 由一个长度为 l 的 bucket 数组组成，每个 bucket 有数量为 b 的存储单元（也被称为槽位），每个槽位存储一个固定长度的元素指纹。每个存储元素有两个候选存储位置：

  • 第一个存储位置直接通过 hash 计算得到；
  • 第二个存储位置由第一个存储位置和元素指纹的 hash 结果进行异或运算得到，该过程不依赖元素的完整信息；

元素会存储在空闲的候选位置中，若两个候选位置都已存有元素，则从 bucket 中随机选择其中一个元素将其存到它的另一个候选位置，然后再将当前元素插入。删除操作会从元素的两个候选位置中找到元素指纹，然后将其删除即可，不会影响到其他元素。
DCF 由同构的 CF 链接而成，初始状态只有一个 CF，当这个初始 CF 写满之后会链接上一个新的同构 CF 作为活跃 CF 来接收元素插入操作，另外 DCF 还支持紧凑算法来将稀疏 CF 中的元素转移到紧凑的 CF 中，从而提高 DCF 的空间效率，而且该算法的执行时间和频率是可控的。

==所以，在我们可以选择更加高效且经济的检索结构来代替传统的 BF。==

负载特征分析

键值存储系统的实际工作负载不仅是读写混合的，而且还具有很明显的数据访问倾斜，这意味着其中占比很小的键值对可能会收到大部分的访问请求。
从 LSM-tree 的数据分布来看，每个 level 的容量都是上一个 level 的好几倍，在 leveldb 中是 10 倍。在一些常见工作负载中，最后一个 Level 大概包含 70%的 SSTable，但是它们只接受 9%的请求。

==所以，可以针对 LSM-tree 的结构特点和实际工作负载特征对检索结构进行针对性设计和优化。==

预期成果与挑战

  • 设计一个使用 Dynamic Cuckoo Filter（DCF）来代替 Bloom Filter 的 LSM-tree 键值存储系统；
  • 根据 LSM-tree 键值存储系统的结构特点和工作负载特征对 DCF 进行结构改进和参数优化；
  • 保证以较低的开销去动态维护 DCF 的结构和功能，实现最优的查询效率；
  • 对整体设计和实现方案进行性能分析和评估；
