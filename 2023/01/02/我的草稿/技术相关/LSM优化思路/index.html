关于 Compaction

Compaction 带来的影响：

  • 写放大，Compaction 需要重写大量的数据；
  • CPU 和带宽竞争，后台执行 Compaction 需要长时间占用 CPU 和 IO 带宽，这会影响前台的读写操作，并影响 L0-L1 的 Compaction 操作，导致 L0 中 SSTable 的积累，进而出现 Write Stall；
  • 缓存失效，执行完毕 Compaction 后会导致 Block Cache 中缓存的 Block 失效；

    关于 KV 分离

    KV 分离后的问题：
  • 垃圾的高效回收，以及回收过程中如果需要读和重写 LSM 树，则会影响前台操作，新数据沉到垃圾回收回写的数据下面的问题
  • 缓存低效
  • Scan 性能低
  • Value Size 较小的情况

    加快检索的速度

    随着存储器件的进步以及缓存的优势（当 workload 存在明显的热数据时，有效的缓存设计能保证大部分需要查询的数据已经加载到了主存中），此时已经很少需要从磁盘中加载数据，在主存中检索数据成为了新的性能瓶颈。
    Bourbon 和 Chucky 这两篇论文都立足于这个背景，Bourbon 是通过 Learned Index 来加快检索 Data Block 的速度，Chucky 是通过设计新的 BF 来加快检索 BF 的时间。

    工作负载

更新密集型工作负载存在的问题

  • WAL 比 MemTable 更快达到容量上限，进而导致 MemTable 在并未达到容量上限时就需要执行 Flush 操作。
  • 高热度的 key 分布在多个 level 中，导致相关的 level 发生级连 Compaction（解决方案：将更新频繁也即高热度的 key 保留在当前 level 中，避免频繁 Compaction 到 next level

删除密集型工作负载存在的问题

  • 被标记删除的 KV 项只有 Compaction 到最后一个 Level 时才会被丢弃，这会导致长时间的空间放大（解决方案：优先选择 SST 中标记删除的 KV 项占比高的执行 Compaction 操作，尽快释放其所占用的磁盘空间

读密集型工作负载存在的问题

  • 读放大
  • 缓存频繁交换导致缓存命中降低

写密集型工作负载存在的问题

  • 写放大
  • 频繁 Compaction 导致 IO 带宽和 CPU 资源的竞争
  • 频繁 Compaction 导致缓存中的 block 都是执行 Compaction 时读取的Block

关于写停滞和写放大

  • SILK：划分 Compaction 的优先级，L0-L1 以及 Flush 操作优先执行，并且拥有抢断权限
  • SLOT：尽量公平分配执行 Compaction 的次数或者说是频率
    如果 L0 和 L1 同时满了，同时出现了 L0-L1 以及 L1-L2 的 Compaction 需求：
  • 先执行 L1-L2 的 Compaction 的话可能会导致 L0 中 SST 的累积，带来写停滞
  • 先执行 L0-L1 的 Compaction 的话，因为 L1 的 SST 数量已经超额，会增加 L1 中参与 L0-L1 的 Compaction 的 SST 的数量，带来更高的写放大
    思路：若出现上述情况，也即 L0 和 L1 都出现 Compaction 的需求，则三层一块执行 Compaction，这样会增加 Compaction 的执行时间（也即会导致 L0 中的 SST 数量没有及时减少，带来写停滞），但能减少写放大。
    SILK 的实验环节并没有展示其设计是否影响了写放大，可以做实验对比 SILK 和 RocksDB 的写放大情况。

实验结论：

  • SILK 确实带来了更高的写放大，且虽然确实减少了 Write Stall，但却怎加了执行时间，换言之，写性能可能更稳定了，但只是稳定的更慢了（关于 write stall 还需要继续做实验验证，数据量太小体现不出来）

    关于多层 BF 确定 KV 项所在的 SST

    问题：
  • 在 leveldb 中，index block 和 filter block 是在打开 SST 文件后放在 table cache 里面的（如果我们把 Table Cache 开到最小就能体现出我们的性能，但这是不公平的）
  • 在默认 4MB 的 SST 大小的情况下，几乎所有的 KV 项在每一个 level 都有一个覆盖其的 SST，导致需要将所有的 KV 项都存放在我们设计的内存多层 BF 中，进而导致内存的多层 BF 膨胀到和直接将所有 SST 的多层 BF 加入到内存再无区别
  • 多层 BF 的删除问题，如何从 BF 中删除已添加的元素是个严重的问题，若使用 CBF 则会大大增加内存开销
  • 同等数据量下 KV 项大小对内存开销的影响

出路：

  • 要么能找出一种新的方法，而不是直接将每层所有 SST 的 BF 聚合为一个 BF 放到内存
  • 要么找出聚合为多层 BF 带来的额外好处
  • 多层 BF 每层采取的 bits-per-key 不同，依次减少，从而节省内存开销

关于布隆过滤器：

  • b：bits-per-key
  • k：hash 函数的个数
  • 当 $k = ln2 \cdot b$，误报率可以表示为 $0.6185^b$

    缓存问题

    缓存失效问题：
  • 及时逐出失效的缓存块 (AC-Key 考虑到了这点)
  • 或者将失效的缓存块替换为新的缓存块
    将 index block 和 filter block 缓存在 table cache ，还是缓存在 block cache，若缓存在 block cache 中那么如何区分 index、filter、data 的缓存优先级和先后顺序的问题。
    AC-key 采用了 block cache，KV cache 以及 KP cache，KP 中指针的格式为<SST ID, Block Offset>

选择 SST 执行 Compaction

现在同时只有单策略的方法，能不能同时具备多种选择策略，然后每次 compaction 时根据权重选择最优的选择策略。
